01 / Retrieval path

Follow the evidence before the answer.

The central design decision is chain of custody: show how a question produces graph context and dense-search evidence before either path reaches synthesis.

02 / Architecture

Two retrieval paths, one inspectable prompt.

The system diagram and implementation split dense semantic retrieval from explicit graph context, then keep both visible at synthesis time.

Interface
React + TypeScript keeps the conversation, source excerpts, topology, and trace in one workspace.
API
FastAPI validates query boundaries and orchestrates retrieval and synthesis.
Vector path
SentenceTransformers and FAISS retrieve semantically related Wikipedia passages.
Graph path
spaCy entity extraction and DBpedia SPARQL add explicit entity and relationship context.
Synthesis
OpenAI produces a concise answer over the evidence that survives retrieval.
03 / Evidence contract

Clear about what is—and is not—running.

A static walkthrough is useful only if it does not impersonate a hosted retrieval service or turn a fixture into a benchmark.

What the recording establishes

The deterministic fixture makes the interface and source-selection interaction reproducible without credentials or multi-gigabyte retrieval artifacts.

What this GitHub Pages site does not host

It does not run FastAPI, FAISS, DBpedia queries, OpenAI synthesis, or the full corpus. Those remain available through the repository's local and Docker setup.

05 / Limitations

Inspectability is not correctness.

Making evidence visible improves reviewability, but it does not remove retrieval, source, or model limitations.

  1. Retrieval quality depends on corpus coverage, chunking, and similarity thresholds.
  2. DBpedia can return sparse context for ambiguous or uncommon entities.
  3. Visible evidence makes the answer easier to inspect; it does not guarantee factual correctness.
  4. The included demo is a deterministic product fixture, not an evaluation of the full retrieval pipeline.
